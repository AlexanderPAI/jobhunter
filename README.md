# КарьеРадар

Персональная система карьерной навигации для поиска и первичной фильтрации вакансий по резюме кандидата.

Версия: `0.3.1`

## Что делает проект

КарьеРадар принимает резюме, извлекает из него профиль кандидата, формирует поисковый запрос, сканирует вакансии на hh.ru и Хабр Карьере, а затем фильтрует найденные возможности под опыт и ожидания кандидата.

Профили, история подборов, найденные вакансии и результаты фильтрации сохраняются в PostgreSQL и изолированы между пользователями. После входа пользователь может просматривать свои сохранённые профили, повторять подбор и экспортировать последнюю выдачу в CSV. Для LLM-агентов можно выбрать GigaChat или OpenRouter.

Версия `0.3.1` закрепляет новое название и визуальный язык продукта. Интерфейс построен вокруг метафоры карьерного радара, а направления дальнейшего развития описаны в [ROADMAP.md](ROADMAP.md).

## Возможности

- Загрузка резюме через веб-интерфейс Streamlit.
- Вход по логину и паролю с JWT-авторизацией.
- Изоляция профилей и подборов между пользователями.
- Выбор LLM-провайдера: GigaChat или OpenRouter.
- Поддержка файлов `.pdf`, `.docx`, `.doc`, `.txt`.
- Автоматическое извлечение профиля кандидата из резюме.
- Генерация поискового запроса для вакансий.
- Поиск вакансий на hh.ru.
- Поиск вакансий на Хабр Карьере.
- Объединение результатов hh.ru и Хабр Карьеры в единую выдачу.
- Удаление дублей по ссылкам.
- LLM-фильтрация вакансий по профилю кандидата.
- Сохранение профилей и истории подборов в PostgreSQL.
- Просмотр списка сохранённых профилей.
- Отдельная страница полного профиля с последним результатом поиска.
- Отображение даты и времени последнего подбора.
- Повторный подбор вакансий для сохранённого профиля.
- Просмотр результата в таблице внутри UI.
- Экспорт итоговой выдачи в CSV без промежуточного хранения CSV на backend.

## Основной пользовательский сценарий

1. Пользователь входит в веб-интерфейс по логину и паролю.
2. Загружает резюме.
3. Нажимает `Анализировать резюме`.
4. Система показывает профиль, краткое описание и поисковый запрос.
5. Пользователь нажимает `Подобрать вакансии`.
6. Backend собирает вакансии с hh.ru и Хабр Карьеры.
7. Агент фильтрует вакансии по профилю кандидата.
8. Пользователь смотрит таблицу и скачивает CSV.
9. Профиль и результаты остаются доступны только его владельцу на странице `Профили`.
10. С детальной страницы профиля можно повторить подбор вакансий.

## Переменные окружения

Для работы приложения нужно выбрать LLM-провайдера, указать его ключ, параметры PostgreSQL и JWT-секрет.

Создайте файл `.env` в корне проекта:

```env
OPENROUTER_KEY=your_openrouter_api_key
GIGACHAT_KEY=your_gigachat_authorization_key
GIGACHAT_MODEL=GigaChat
GIGACHAT_URL=https://api.giga.chat/v1/chat/completions
GIGACHAT_VERIFY_SSL_CERTS=false
LLM_PROVIDER=gigachat
POSTGRES_USER=jobhunter
POSTGRES_PASSWORD=change_me
POSTGRES_HOST=db
POSTGRES_PORT=5432
POSTGRES_DB=jobhunter
JWT_SECRET=replace_with_a_long_random_secret_at_least_32_chars
JWT_EXPIRE_MINUTES=480
```

`LLM_PROVIDER` принимает `gigachat` или `openrouter`. Значения `OPENROUTER_KEY`, `GIGACHAT_KEY` и `JWT_SECRET` указывайте без кавычек: некоторые способы передачи `.env` в Docker сохраняют кавычки как часть секрета.

## Запуск через Docker Compose

```bash
docker compose up --build
```

После запуска:

- Frontend: `http://localhost:8501`

Backend и PostgreSQL не публикуют порты на хост. Backend доступен только frontend-контейнеру во внутренней Docker-сети.

## Пользователи и вход

Публичной регистрации нет. После запуска создайте пользователя из backend-контейнера:

```bash
docker compose exec backend python -m backend.create_user username
```

Скрипт интерактивно запросит пароль (минимум 8 символов). После этого пользователь может войти на `http://localhost:8501`. Все страницы интерфейса, API профилей, поисков и фильтрации требуют JWT, а история фильтруется по владельцу.

## Миграции

При запуске через Docker Compose backend автоматически выполняет `alembic upgrade head` после готовности PostgreSQL.

Для ручной работы с миграциями:

```bash
alembic -c backend/alembic.ini revision --autogenerate -m "migration_name"
alembic -c backend/alembic.ini upgrade head
```

## API

Все маршруты профилей, поиска и фильтрации требуют `Authorization: Bearer <token>`.

### Вход

```http
POST /v1/auth/token
```

Принимает `username` и `password` как OAuth2 form data и возвращает JWT access token.

### Анализ резюме

```http
POST /v1/cv_analyzer/send_cv
```

Принимает файл резюме и возвращает:

- `search_prompt` - поисковый запрос для агента подбора вакансий;
- `user_profile` - структурированный профиль кандидата.
- `profile_id` - UUID сохранённого профиля.

### Поиск вакансий

```http
POST /v1/searcher/chat
```

Принимает:

- `message` - текст поискового запроса;
- `profile_id` - UUID профиля кандидата.

Возвращает `search_id` — UUID сохранённого запуска поиска. Вакансии сохраняются в PostgreSQL.

### Фильтрация вакансий

```http
POST /v1/filter/check
```

Принимает `search_id`, загружает профиль и найденные вакансии из PostgreSQL, сохраняет результат LLM-фильтрации и возвращает JSON со списком подходящих вакансий.

## Хранение данных

В PostgreSQL используются пять основных сущностей:

- `users` — учётные записи, хеши паролей и статус активности;
- `candidate_profiles` — структурированные профили кандидатов, исходный JSON и текст резюме;
- `search_runs` — история запусков поиска, запросы, фильтры и статистика;
- `vacancies` — дедуплицированные вакансии из разных источников;
- `search_results` — связь поиска с вакансией, порядок выдачи и результат фильтрации.

Frontend получает профили и историю только через защищённый backend API. Прямого сетевого доступа frontend к PostgreSQL нет.

## Ограничения

- Подбор зависит от качества распознавания резюме и ответа LLM.
- Публичной регистрации нет: учётные записи создаются администратором через CLI.
- Фильтрация вакансий выполняется батчами и может занимать время.
- Парсинг внешних сайтов зависит от доступности hh.ru и Хабр Карьеры.
- Профиль пока нельзя редактировать вручную из UI.
- Загруженные файлы резюме сохраняются локально в `backend/storage/cv`.

### Цель релиза

Главная цель `v.0.1.0` - проверить полный путь от резюме до списка релевантных вакансий:

```text
резюме -> профиль кандидата -> поисковый запрос -> сбор вакансий -> фильтрация -> CSV
```

### Что пока не входит в релиз

- Авторизация пользователей.
- Сохранение истории поисков в базе данных.
- Редактирование профиля кандидата перед поиском.
- Настройка источников вакансий из UI.
- Расширенная аналитика по найденным вакансиям.
- Отправка откликов на вакансии.
- Production-ready деплой и мониторинг.

### Статус релиза

`v.0.1.0` считается MVP-релизом. Он пригоден для локального запуска, демонстрации основной идеи и дальнейшего развития продукта.
